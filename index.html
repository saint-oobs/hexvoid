1. Architecture Overview & Core Philosophy

HexVoid is a custom, highly optimized Discord bot framework built on core non-negotiable pillars:

  • Stateless: Unbounded lists or dictionaries are never cached in RAM. Every process relies purely on async SQLite lookups for memory safety. We scale without memory leaks.
  • SQLite-Driven: Everything from moderation cases to gallery restrictions is stored locally in a high-speed, WAL-enabled SQLite database (database/db.py).
  • Zero Legacy Code: HexVoid uses modern app_commands (Slash Commands) exclusively. No deprecated prefix commands are supported for user-facing modules.
  • Ephemeral Ghost Mode UI: Interactions are designed to be clean and completely invisible. Commands execute silently, and standard bot replies are ephemeral to avoid cluttering the chat with command spam.

2. Working Principles

The AutoMod Strike Ladder and Decay Engine

HexVoid implements an active user punishment ladder built into the core moderation system. As users break rules, they ascend internal strike levels, mapped to escalating automated consequences.

  • Sequential Escalation: Violations auto-escalate users to higher tiers (e.g., Warning, 1hr Mute, 24hr Mute, Ban) depending on their accumulated history in the STRIKE_LADDER config.
  • The Decay Engine: A background periodic task (tasks.loop) continually evaluates user strike epochs. Once an established cooldown duration expires without incident, the engine natively decays the strike level by one step, actively promoting forgiveness for reformed members.
  • Action Verification: Punishments applied via the strike engine natively catch discord.Forbidden exceptions and emit Mod-Log embeds tracking whether the bot lacked permission to complete an escalation.

Memory-Safe UI Views

Discord UI Components (Buttons, Dropdowns, Modals) and persistent tasks require highly defensive memory management.

  • Task GC Tracking: Background tasks holding ephemeral view interactions explicitly maintain strong references (bg task refs) to prevent the Python Garbage Collector from sweeping them out of memory, guaranteeing timeouts/modals execute flawlessly.
  • Severed Database Cursors: Heavy lookups utilizing aiosqlite immediately fetch data, collapse the results into dictionaries containing pure values, and instantly drop/release the cursor reference (e.g., aiosqlite Row -> dict mappings). Database cursors are never held open during interactive UI lifecycles, guaranteeing zero DB lockups or leaked database row generators.

4. Background Tasks & Event Listeners

HexVoid utilizes silent background loops (via tasks.loop) to execute strict cron jobs and maintain data integrity without manual intervention.

  • tickets.py (30 min loop): Sweeps for abandoned or inactive tickets based on inactivity limits and automatically clears them out.
  • reminders.py (12 sec loop): Polls the database for expired personal reminders to send alerts securely.
  • giveaway.py (15 sec loop): Actively monitors live giveaways to trigger automated endings, pick winners, and distribute prizes safely.
  • events.py & scheduler.py: Contains rolling timers for rotating the bot status and processing core scheduled jobs natively.
  • events.py Cache Warmup (on_ready task): A background daemon that slowly reconstructs the internal discord.py message cache (capped at 100,000) post-reboot by silently iterating text and thread channels, ensuring robust delete/edit tracking.
  • autoroles.py (60 sec loop): A reaper cron loop scanning hexvoid_timed_roles for expired entries to strip roles cleanly using Discord API.
  • automod.py (60 sec loop): The primary Automod Decay Engine that assesses rule breakers and actively decays strike levels over time for reformed members.
  • appeals_portal.py & Support Tickets: Detects incoming webhook embeds and attaches interactive Persistent Views (Appeals vs Support). Redis Upstash caching is natively utilized.
  • audit_log.py (Event Listener): Hybrid audit logging system built on pure event-driven pipelines using Attribution dicts and TTL dedup cache logic to avoid excessive API polling.

5. Database Schema Reference

The database/db.py layer handles HexVoid's heavy lifting. Below are the critical tables actively powering the bot's state:

  • cases & user_strikes: The heart of HexVoid's moderation. Tracks active/historic moderation cases, moderator IDs, decay timers, and dynamic strike levels per user.
  • active_punishments: Tracks time-based mutes/bans natively using scheduled expiry checkpoints instead of RAM timeouts.
  • giveaways & giveaway_entries: Houses full giveaway logic including requirements, whitelists, and live participant weights.
  • ticket_panels & active_tickets: Dedicated tracking for full ticket portal configurations, custom settings, log channels, and individual live state management.
  • reminders & afk_users: Deep state tracking for custom alerts, repetitive DM notification loops, AFK statuses, and snoozes.
  • counting_config & automod_config: JSON-serialized configuration schemas dictating guild-specific preferences explicitly without cache duplication.

6. Developer "Do's and Don'ts" (The HexVoid Standard)

Adhere to the following ruleset:

  • DO: Use the Dual-Check Permission system (Both @app_commands.default_permissions(...) AND @validate_context(...)).
  • DO: Deploy strict Context Decorators (@app_commands.allowed_installs(guilds=True) and @app_commands.allowed_contexts(...)).
  • DO: Defer immediately with await interaction.response.defer(ephemeral=True) for ANY command hitting the API/DB.
  • DO: Wrap all API role/message manipulation in try/except (discord.Forbidden, discord.HTTPException).
  • DO: Wrap the inside of ANY tasks.loop in a broad try/except Exception so the daemon loop never dies silently.
  • DON'T: Cache unbounded lists/dictionaries in RAM. Rely purely on async SQLite lookups for lookup validation.
  • DON'T: Pop a discord.ui.Modal AFTER an interaction has been deferred. Modals must be pushed on the initial response frame.
